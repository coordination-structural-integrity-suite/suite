# Software Specification Precision Audit
Use when auditing a software specification — application programming interface contract, requirements document, technical design document, or schema definition — for requirements that can't actually be tested by an independent engineer. Software specifications have characteristic precision failures: acceptance criteria that depend on the author's intent rather than observable behavior, loosely typed definitions, error handling that doesn't distinguish error classes, and technical patterns transplanted from other systems without documenting whether the structural conditions match. The prompt finds those failures and returns a precision deficit map organized by failure type, with the most likely sources of bugs or integration failures first.

You are auditing the attached software specification using the Precision-First Design Standard (CC BY 4.0, Regis Chapman). Precision fails in two directions: a precision deficit is any requirement, definition, or interface that cannot be evaluated by an independent tester using only the specification and observable system behavior; a precision imposition is any specification detailed enough to pass acceptance testing while the actual behavioral outcome the specification was designed to ensure remains absent. Both require correction.

Check for these nine common precision failures in software specifications:

1. Untestable acceptance criteria: requirements that depend on the original author's intent rather than observable behavior (e.g., "the system should handle errors gracefully," "provide a good user experience," "respond in a timely manner"). For each one found, propose a criterion an independent tester can evaluate without access to the author.
2. Loosely typed definitions: places where a string type is used where a typed enum is possible, where an "other" or "miscellaneous" category exists without an overflow procedure, or where a data format is described in prose rather than in a schema. For each one found, propose the precise type.
3. Unclassified error handling: error responses that use generic categories ("unexpected error," "internal server error," "something went wrong") without distinguishing error classes by structural signature, detectability, and recovery procedure. For each one found, propose a classification.
4. Ambiguous state boundaries: places where the specification does not clearly distinguish between states that the system treats differently (e.g., "active" vs. "enabled," "pending" vs. "processing," "failed" vs. "error"). For each one found, propose operational definitions that make the boundary falsifiable.
5. Interface contracts with implicit assumptions: API endpoints, event schemas, or data contracts that assume context the consumer may not have, or that leave behavior unspecified for valid inputs (e.g., "returns a list of results" without specifying ordering, pagination, empty-set behavior, or maximum size). For each one found, list the unspecified behaviors and propose defaults.
6. Inaction invisibility: detection mechanisms, alerting systems, or error reporting pipelines that produce outputs but provide no way for an independent observer to determine whether those outputs were acted on, without the cooperation of the responsible party. For each one found, propose a specification that makes inaction visible without requiring actor cooperation.
7. Single-mechanism critical properties: critical system properties (e.g., authentication, data integrity, access control, rate limiting) that depend on a single protective mechanism with no structurally independent backup. For each one found, identify the critical property, name the single mechanism protecting it, and propose at least one structurally distinct backup mechanism whose failure would be detectable independently.
8. Baseline-free assessment: two forms. First, detection outputs or classification systems that record system states without specifying a required contextual baseline, where identical measurement outputs could indicate structurally distinct conditions requiring different responses. A detection system that classifies without a declared reference baseline has a precision deficit in the instrument: correct readings for one system context produce incorrect response prescriptions for another. For each one found, specify what baseline context must be declared before the assessment architecture is interpretively precise. Second, technical patterns or design mechanisms adopted from a different system context without documenting the structural conditions they were designed and applied in, the structural conditions of the adopting system, and either a congruence confirmation or calibration adjustments. A pattern transplanted intact from a different system architecture carries an undocumented assumption that origin-context structural conditions obtain; when that assumption is false, the pattern may be well-specified internally while failing to address the actual failure modes present in the adopting system.

9. Precision imposition: specifications detailed enough to pass acceptance testing while the actual behavioral outcome the specification was designed to ensure remains absent. Characteristic signature: all specified outputs can be produced, but the intended user or system outcome is not achieved. For each one found, identify what the specification was intended to achieve and propose a criterion that tests the actual behavioral outcome rather than the mechanical output.

Output the results as a precision map organized by failure type, with the failures most likely to cause bugs, disputes, or integration failures first. Label each finding as a precision deficit (under-specification) or precision imposition (over-specification) so that the corrective direction is clear.

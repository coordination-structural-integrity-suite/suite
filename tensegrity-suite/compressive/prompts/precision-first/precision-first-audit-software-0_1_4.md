# Software Specification Precision Audit
Use when auditing a software specification — application programming interface contract, requirements document, technical design document, or schema definition — for requirements that can't actually be tested by an independent engineer. Software specifications have characteristic precision failures: acceptance criteria that depend on the author's intent rather than observable behavior, loosely typed definitions, error handling that doesn't distinguish error classes, and technical patterns transplanted from other systems without documenting whether the structural conditions match. They also fail in the opposite direction: specifications detailed enough to pass all acceptance testing while the actual behavioral outcome the specification was designed to ensure remains absent. The prompt finds both kinds of failures and returns a precision map organized by failure type, with the most likely sources of bugs, disputes, or integration failures first.

You are auditing the attached software specification using the Precision-First Design Standard (CC BY 4.0, Regis Chapman). Precision fails in two directions: a precision deficit is any requirement, definition, or interface that cannot be evaluated by an independent tester using only the specification and observable system behavior; a precision imposition is any specification detailed enough to pass acceptance testing while the actual behavioral outcome the specification was designed to ensure remains absent. Both require correction.

Check for these eight precision failure types in software specifications. Each type runs in two directions: the floor condition asks whether the specification under-specifies; the ceiling condition asks whether it over-specifies in a way that makes the intended behavioral outcome unreachable or untestable.

1. Untestable acceptance criteria. Floor: requirements that depend on the original author's intent rather than observable behavior (e.g., "the system should handle errors gracefully," "provide a good user experience," "respond in a timely manner"). For each one found, propose a criterion an independent tester can evaluate without access to the author. Ceiling: Are any criteria specified in such mechanical detail that they test system output rather than behavioral outcome — a system can pass every criterion while the intended user or system outcome remains absent?

2. Loosely typed definitions. Floor: places where a string type is used where a typed enum is possible, where an "other" or "miscellaneous" category exists without an overflow procedure, or where a data format is described in prose rather than in a schema. For each one found, propose the precise type. Ceiling: Does any type taxonomy create more categories than the data has distinguishable states, or require classification before the information needed to classify is available — forcing real inputs into categories that do not fit?

3. Unclassified error handling. Floor: error responses that use generic categories ("unexpected error," "internal server error," "something went wrong") without distinguishing error classes by structural signature, detectability, and recovery procedure. For each one found, propose a classification. Ceiling: Does any error classification system create so many overlapping categories that real errors fit multiple boxes simultaneously, or fit none — making the classification less useful than a simpler system would be?

4. Ambiguous state boundaries. Floor: places where the specification does not clearly distinguish between states that the system treats differently (e.g., "active" vs. "enabled," "pending" vs. "processing," "failed" vs. "error"). For each one found, propose operational definitions that make the boundary falsifiable. Ceiling: Are any state definitions specified so narrowly that they eliminate false positives by systematically excluding the actual system condition the boundary was meant to capture?

5. Interface contracts with implicit assumptions. Floor: API endpoints, event schemas, or data contracts that assume context the consumer may not have, or that leave behavior unspecified for valid inputs (e.g., "returns a list of results" without specifying ordering, pagination, empty-set behavior, or maximum size). For each one found, list the unspecified behaviors and propose defaults. Ceiling: Are any contracts specified so completely that conforming implementations cannot serve real use cases — compliance is achievable but the behavioral outcome the contract was designed to guarantee is not?

6. Inaction invisibility. Floor: detection mechanisms, alerting systems, or error reporting pipelines that produce outputs but provide no way for an independent observer to determine whether those outputs were acted on, without the cooperation of the responsible party. For each one found, propose a specification that makes inaction visible without requiring actor cooperation. Ceiling: Does any inaction detection or alerting mechanism fire on ordinary operational delays, producing outputs so frequently that they become indistinguishable from normal system behavior?

7. Single-mechanism critical properties. Floor: critical system properties (e.g., authentication, data integrity, access control, rate limiting) that depend on a single protective mechanism with no structurally independent backup. For each one found, identify the critical property, name the single mechanism protecting it, and propose at least one structurally distinct backup mechanism whose failure would be detectable independently. Ceiling: Do any overlapping protective mechanisms contradict each other, generate false positive alerts, or make operating the system structurally unachievable under normal conditions?

8. Baseline-free assessment. Floor: two forms. First, detection outputs or classification systems that record system states without specifying a required contextual baseline, where identical measurement outputs could indicate structurally distinct conditions requiring different responses. A detection system that classifies without a declared reference baseline has a precision deficit in the instrument: correct readings for one system context produce incorrect response prescriptions for another. For each one found, specify what baseline context must be declared before the assessment architecture is interpretively precise. Second, technical patterns or design mechanisms adopted from a different system context without documenting the structural conditions they were designed and applied in, the structural conditions of the adopting system, and either a congruence confirmation or calibration adjustments. Ceiling: Does any baseline requirement impose conditions that cannot be satisfied in real deployment, or that require data collection that cannot be completed before the detection it was meant to enable — making the baseline requirement itself the blocker?

Output the results as a precision map organized by failure type, with the failures most likely to cause bugs, disputes, or integration failures first. Label each finding as a precision deficit (under-specification) or precision imposition (over-specification) so that the corrective direction is clear. For each finding, look to what the specification itself says about the parties it is written for — consuming services, implementing engineers, end users, operators, or other named roles. Where the specification names those parties, use that identification to note who cannot implement correctly, predict system behavior, or verify behavioral outcomes (precision deficit), or who must demonstrate compliance in ways that do not test the intended outcome (precision imposition). Where the specification does not identify its intended parties, note the absence as a structural observation rather than a gap to supply independently.

Corpus evolution note: if this audit is being conducted as part of a corpus evolution question — evaluating whether a proposed new technical standard, specification element, or document should be added to the corpus — apply the transclusion check before beginning the precision audit. Ask: is there a frame at which an existing element is already true, such that the original precision and the proposed requirement are both fully operative within it as distinct instantiations, with neither requiring the other to diminish? If yes, the transclusion is the correct move and no new precision-complete element is needed. If no such frame exists, proceed with this audit.
